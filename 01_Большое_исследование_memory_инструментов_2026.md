# Большое исследование memory-инструментов 2026

Дата: 2026-04-13
Статус: draft / market research for `НоваяПамятьV3`

## 1. Executive summary

Короткий вывод:

> Для OpenClaw/Echo сейчас выигрывает **не один memory-фреймворк**, а слоистая связка:  
> **Obsidian/markdown canon + SESSION-STATE + QMD/hybrid retrieval + local embeddings + simple governance + domain adapters.**

### Лучший выбор сейчас

1. **Obsidian + markdown canon** — главный дом памяти.
2. **Karpathy / LLM Wiki pattern** — форма памяти: raw → curated → compiled wiki.
3. **SESSION-STATE.md + daily summaries** — hot/working memory.
4. **WikiLinks / MOCs / Properties / Bases** — навигация и лёгкая структурность.
5. **QMD + hybrid search** — поиск по vault.
6. **Local embeddings** — default для semantic retrieval.
7. **ADR / Known_Bad / Nightly Librarian-style maintenance** — гигиена и анти-повторы.

### Что вводить рано

- SESSION-STATE.md;
- raw/daily/curated/canon слои;
- минимальные frontmatter/Properties;
- WikiLinks и страницы входа;
- QMD как helper;
- BM25 + vector как принцип поиска;
- limited memory slices для подагентов;
- draft-first для Telegram/публикаций.

### Что вводить позже

- LCM/Lossless Claw или ClawMem как continuity/runtime pilot;
- Basic Memory/OpenMemory/ClawMem как markdown-agent memory pilot;
- ByteRover/Hindsight/Mengram для coding/procedural agents;
- Mem0/Zep/Letta/LangMem/MemOS/MemMachine/MemU как agent-native candidates;
- Graphify/Graphiti/Cognee/LightRAG/GraphRAG как derived advanced layer.

### Что не брать как core

- любой инструмент, который забирает source of truth из markdown в непрозрачную базу;
- тяжёлый graph-first stack на фазе 0;
- cloud persistence как сердце локально закрытой Эхо;
- “SOTA по маркетингу” без проверки на твоём vault.

### Где лучше своё простое решение

- SESSION-STATE.md;
- ADR/;
- Known_Bad/;
- minimal Properties/frontmatter;
- WikiLinks/MOCs;
- daily digest;
- Telegram draft-first gates;
- visual/artifact cards;
- vault librarian checklist.

### Где готовое решение сильнее

- retrieval/search engine;
- embeddings inference/API;
- vector DB/index;
- graph memory, если понадобится;
- advanced agent memory backend;
- coding-agent memory, если ByteRover/Hindsight реально окажутся удобнее.

## 2. Карта категорий

| Категория | Что входит | Решение сейчас | Почему |
|---|---|---|---|
| hot memory / session-state | SESSION-STATE.md, short active task state | Делаем сами сразу | Это простая идея, не зависимость. |
| session continuity | LCM/Lossless Claw, ClawMem, Hipocampus | Пилот позже | Сначала markdown+SESSION-STATE, потом проверка на долгих задачах. |
| vault retrieval | QMD + hybrid search | Брать сейчас | QMD ищет, но не является source of truth. |
| markdown/wiki memory | Karpathy/LLM Wiki + Obsidian + WikiLinks/MOCs | Брать сейчас как форму | Это основа цифрового мозга. |
| Obsidian overlays | Properties, Bases, Daily notes | Брать сейчас | Built-in/простые практики, низкая цена. |
| graph memory | Graphify, Graphiti, Cognee, GraphRAG | Брать позже | Граф — derived layer после чистого канона и retrieval. |
| agent-native memory | Mem0, Zep, Letta, LangMem, MemOS, MemMachine, MemU | Пилоты позже | Полезно для Эхо/подагентов, но опасно как ранний core. |
| personal/user memory | MemMachine ideas + our User/Diary layer | Сначала своё markdown, потом backend | Личные truths требуют прав, provenance и осторожного promotion. |
| visual/artifact memory | Visual taste cards, Comfy cards, later multimodal embeddings | Делаем свои карточки сейчас; tooling позже | Нужен доменный канон вкуса, а не общий граф с картинками. |

| Telegram/social memory | свой Telegram bridge + RSSidian-like ideas | Делаем адаптер сами | Из-за прав публикации и prompt injection нельзя отдавать core внешнему тулу. |
| maintenance/librarian | Nightly Librarian, summaries, stale checks | Делаем сами сначала | Это гигиена vault, не отдельный монстр. |
| security / prompt injection / rights | rights model, draft-first, limited slices | Брать как обязательный L0 | Без этого Эхо может быть живой, но руки должны быть короткими. |
| presentation/explanation layer | NotebookLM phase decks later | Отложить до утверждения roadmap | Сначала решения, потом презентации. |

## 3. Балльная шкала

Каждый инструмент оценивается по 10 критериям, 0–5 за каждый, максимум 50.

| # | Критерий | Что означает |
|---:|---|---|
| 1 | Польза для цели Эхо/Джарвис | насколько прямо помогает local assistant, vault, Telegram, Comfy, подагентам |
| 2 | Зрелость проекта/подхода | активность, docs, issues, adoption, понятность формата |
| 3 | Простота внедрения | сколько риска и настройки нужно для первого рабочего эффекта |
| 4 | Local-first | можно ли держать память локально и приватно |
| 5 | Obsidian/markdown совместимость | не ломает ли .md canon и человекочитаемость |
| 6 | Заменяемость | можно ли вынуть инструмент через адаптер |
| 7 | Качество поиска/памяти | реальный recall/continuity/synthesis, а не маркетинг |
| 8 | Безопасность | права записи, prompt injection surface, публикация наружу |
| 9 | Поддерживаемость | можно ли обслуживать самому и пережить апдейты GitHub |
| 10 | Fit для Эхо/подагентов | поддерживает главный агент + ограниченные сестрички |

Важно: балл — не “истина”, а **рабочий фильтр**. Для OpenClaw важны не только звёзды GitHub, но и local-first, markdown/Obsidian fit, заменяемость, безопасность прав и жизнь с Эхо/сестричками.

## 6. Важные наблюдения

### 6.1. QMD, LCM, OpenViking, ClawMem — не одно и то же

- **QMD** — фонарь/каталог: ищет по vault.
- **LCM/Lossless Claw** — continuity-компрессор: помогает не терять смысл при длинных задачах.
- **ClawMem** — runtime memory layer с hybrid RAG и MCP/agent integration.
- **OpenViking** — тяжёлая надсистема управления контекстами.

Они могут дополнять друг друга, но в фазе 0 нужен не весь набор.

### 6.2. Karpathy / LLM Wiki — это форма, а не dependency

От Карпаты берём:

- raw → curated;
- compiled wiki pages;
- index pages;
- links между знаниями;
- human-readable markdown;
- maintenance loop.

Не берём как “установить и забыть”. Это рабочий стиль ведения мозга.

### 6.3. Graph — поздний слой

Graphify/Graphiti/Cognee/GraphRAG полезны, когда уже есть:

- чистый canon;
- нормальные связи;
- стабильный retrieval;
- понятные entity pages;
- вопрос, на который обычный search уже не отвечает.

Если начать с графа, получится красивая карта болота.

### 6.4. Agent-native memory нужна, но не первой

Mem0/Zep/Letta/LangMem/MemOS/MemMachine/ByteRover/MemU важны, но они решают **runtime/agent backend** задачи.

Для Эхо это может быть полезно позже, но сейчас риск такой:

- уведут truth в свой storage;
- усложнят права;
- дадут lock-in;
- будут конфликтовать с Obsidian canon.

Поэтому правильный путь: сначала canon + retrieval + rights, потом пилоты через adapters.

### 6.5. Embeddings — не религия

Qwen/Gemini/OpenAI/local embeddings конкурируют как провайдеры/модели, но архитектура должна быть такой:

```text
markdown canon -> chunking/index adapter -> embedding provider -> vector index -> retrieval/rerank -> prompt context
```

Если модель меняется, канон не меняется.

## 8. Ещё три важных материала, которые усиливают V3

### 8.1. Karpathy `llm-wiki`

Источник: [gist](https://gist.github.com/karpathy/442a6bf555914893e9891c11519de94f)

Что особенно важно для нас:

- wiki как **compiled knowledge**, а не одноразовый RAG;
- 3 слоя: **raw sources -> wiki -> schema**;
- операции: **ingest / query / lint**;
- `index.md` и `log.md` как отдельные навигационные файлы;
- `QMD` упомянут как хороший local search option, когда wiki вырастает.

Ключевая мысль для V3:

> Канон должен жить в markdown/wiki, а поиск и автоматизация должны стоять рядом, а не вместо него.

### 8.2. Rohit `LLM Wiki v2`

Источник: [gist](https://gist.github.com/rohitg00/2067ab416f7bbe447c1977edaaa681e2)

Что он добавляет поверх Карпаты:

- **memory lifecycle**: working -> episodic -> semantic -> procedural;
- **supersession**: новая версия знания должна явно помечать старую как устаревшую;
- **retention/forgetting**: не всё должно жить в одинаковом приоритете вечно;
- **typed graph layer** поверх страниц;
- **hybrid search** как BM25 + vectors + graph traversal;
- **hooks / automation / quality scoring / audit trail**.

Ключевая мысль для V3:

> Карпаты даёт форму, а Rohit даёт механизм, как эта форма живёт долго и не превращается в свалку.

### 8.3. `andrej-karpathy-skills`

Источник: [GitHub](https://github.com/forrestchang/andrej-karpathy-skills)

Это не memory-engine и не wiki-tool. Это важный **поведенческий слой для агента**, который работает с памятью.

Четыре принципа, которые нам полезны:

- **Think Before Coding** — не делать молча неверные предположения;
- **Simplicity First** — не раздувать архитектуру;
- **Surgical Changes** — не трогать соседние вещи без нужды;
- **Goal-Driven Execution** — работать от критериев готовности и проверки.

Ключевая мысль для V3:

> Даже хорошая память портится, если агент, который её ведёт, слишком самоуверенный, шумный и любит городить лишнее.

### Итог по этим трём материалам

Если совсем по-простому:

- **Karpathy** = форма wiki-памяти;
- **Rohit v2** = как дать wiki жизненный цикл, hooks и quality control;
- **karpathy-skills** = как дисциплинировать самого агента, который эту wiki обслуживает.

Это не три конкурента, а три слоя одной идеи.
